# Антонова Мария
# Описание
База данных предназначена для автоматизации процесса заказа клиентами наличных денежных средств в банке с последующим получением в отделении. Система хранит информацию о клиентах и их счетах, отделениях и сотрудниках, заказах, а также сопутствующих процессах, связанных с  доставкой денежной средств и транзакциями. 
# Функциональные требования к системе в части учета данных
1. Управление клиентскими данными
FR1.1 Система должна обеспечивать хранение и актуализацию персональных данных клиентов (ФИО, контактные данные, паспортные данные)
FR1.2 Система должна поддерживать различные статусы клиента ("активен", "заблокирован", "удален") с возможностью изменения статуса
FR1.3 Система должна автоматически фиксировать дату создания и дату обновления записи о клиенте
FR1.4 Система должна обеспечивать поиск клиента по параметрам (ФИО, телефон, email, номер счета)
2. Учет банковских счетов
FR2.1 Система должна хранить информацию о всех счетах клиента с указанием валюты и типа счета (дебетовый/кредитный)
FR2.2 Система должна отслеживать статус счета ("активен", "заморожен", "закрыт")
FR2.3 Каждый счет должен быть привязан к конкретному клиенту
FR2.4 Система должна обеспечивать получение актуальной информации о счетах из АБС банка
3. Обработка заказов денежных средств
FR3.1 Система должна создавать запись о заказе наличных с уникальным идентификатором
FR3.2 Система должна фиксировать все ключевые параметры заказа: сумму, валюту, выбранное отделение, дату запроса
FR3.3 Система должна отслеживать жизненный цикл заказа через систему статусов
FR3.4 Система должна генерировать уникальный код получения для каждого заказа
FR3.5 Система должна назначать ответственного сотрудника на заказ и фиксировать сотрудника, выполнившего выдачу
FR3.6 Система должна хранить историю изменения статусов каждого заказа
4. Учет транзакций
FR4.1 Система должна регистрировать все транзакции, связанные с заказами наличных
FR4.2 Каждая транзакция должна содержать информацию о типе (списание/возврат), сумме, валюте и статусе
FR4.3 Система должна поддерживать механизм отката транзакций (через связь с откатной транзакцией)
FR4.4 Система должна фиксировать ошибки при выполнении транзакций для последующего анализа
5. Управление лимитами
FR5.1 Система должна поддерживать различные типы лимитов для клиентов (дневные, месячные, максимальная сумма заказа)
FR5.2 Система должна отслеживать текущее использование лимитов
FR5.3 Система должна автоматически сбрасывать периодические лимиты по истечении периода
FR5.4 Система должна проверять соблюдение лимитов при создании нового заказа
6. Проверка безопасности 
FR6.1 Система должна проводить проверки безопасности для каждого заказа
FR6.2 Система должна поддерживать различные типы проверок (фрод-мониторинг, ручные проверки)
FR6.3 Система должна фиксировать результаты проверок и назначенных сотрудников безопасности
FR6.4 Система должна хранить причины автоматической блокировки заказов
7. Уведомления
FR7.1 Система должна отправлять уведомления клиентам о статусе их заказов
FR7.2 Система должна отправлять уведомления сотрудникам о новых заявках и необходимости выполнения действий
FR7.3 Система должна поддерживать различные каналы доставки уведомлений (email, SMS, push, внутренние уведомления)
FR7.4 Система должна отслеживать статус доставки и прочтения уведомлений
8. Учет доставки денежных средств
FR8.1 Система должна поддерживать информацию о доставке наличных в отделения (если применимо)
FR8.2 Система должна фиксировать данные инкассаторской службы, транспорта и ответственных лиц
FR8.3 Система должна отслеживать статус доставки и подтверждение получения
9. Работа с отделениями и сотрудниками
FR9.1 Система должна хранить актуальную информацию об отделениях банка (адрес, контакты, часы работы, статус)
FR9.2 Система должна вести учет сотрудников с привязкой к отделениям
FR9.3 Система должна поддерживать ролевую модель доступа для сотрудников
FR9.4 Система должна отслеживать статус сотрудников ("активен", "в отпуске", "уволен")
10. История заказов
FR10.1 Система должна хранить полную историю изменения статусов заказов
FR10.2 Система должна фиксировать, кто и когда вносил изменения (сотрудник или система)
FR10.3 Система должна сохранять техническую информацию об изменениях (IP-адрес)
FR10.4 Система должна регистрировать все неудачные попытки создания заказов с указанием причины
